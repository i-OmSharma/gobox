The Configuration & Path Logic

What it does: Defines the "blueprint" for a container's resource limits and calculates where this container lives in the Linux filesystem.
How it works: It creates a struct to hold configuration data. The path() method simply concatenates the root cgroup directory (/sys/fs/cgroup) with the unique Container ID.
Problem Solved: Abstraction. Instead of hardcoding paths like /sys/fs/cgroup/my-container everywhere in your code, you have a single source of truth.
If the root path changes (e.g., in a different Linux distro), you only update one constant.
Core CS Concepts:
    Data Structures: Using a struct to group related data.
    Filesystem Hierarchy: Understanding that Linux resources are represented as directories.

The Apply Function (The Brain)

What/How: Creates the folder /sys/fs/cgroup/<ID>.
Problem Solved: Initialization. In cgroups v2, creating the directory automatically generates the control files (like memory.max) inside it. You can't set limits if the folder doesn't exist.
CS Concept: Virtual Filesystems (sysfs). You aren't creating a real folder on a disk; you are asking the Kernel to create a new resource group object.

Enable Controllers

What/How: Writes +memory +cpu to the parent's cgroup.subtree_control file.
Problem Solved: Security & Delegation (cgroups v2). In modern Linux, child cgroups don't get resource controls by default. 
The parent must explicitly say, "I allow my children to control Memory and CPU." Without this, writing to memory.max would fail with EOPNOTSUPP (Operation not supported).
CS Concept: Access Control Lists (ACLs) / Capability Delegation. This is a permission system where privileges are passed down the hierarchy.

    Set Memory Limit
        1. What/How: Writes the byte limit (e.g., 268435456 for 256MB) to the memory.max file.
        2. Problem Solved: Resource Isolation (Memory). Prevents the "Noisy Neighbor" problem. Without this, one buggy container could consume all host RAM, crashing the entire server.
        3. CS Concept: Kernel Accounting. The kernel tracks every byte allocated by processes in this group. If they exceed the limit, the Kernel triggers the OOM Killer.
    Set CPU Limit
        1. What/How: Writes "50000 100000" (50ms quota per 100ms period) to cpu.max.
        2. Problem Solved: CPU Scheduling/Fairness. Ensures a container doesn't monopolize the CPU. It guarantees that other processes on the host get their turn to run.
        3. CS Concept: Time Slicing & Scheduler Algorithms. The Linux Completely Fair Scheduler (CFS) uses these values to decide how long a process can run before being paused to let others run.
    OOM Group
        1. What/How: Sets a flag that tells the kernel to kill all processes in the group if memory runs out.
        2. Problem Solved: Atomic Failure. If a container has 5 processes and runs out of memory, killing just one might leave the app in a broken state. 
            Killing all of them ensures the container dies cleanly and can be restarted.
        3. CS Concept: Process Group Management. Treating a set of processes as a single atomic unit for lifecycle management. 
    Attach Process
        What/How: Writes the Process ID (PID) into the cgroup.procs file.
        Problem Solved: Enforcement. Limits are useless if no process is subject to them. This moves the specific process into the "jail" we just built.
        CS Concept: Process Association. Linking a running execution context (PID) to a resource policy (Cgroup).

Cleanup block explain
1. The "Ghost Directory" Check
The Issue: If your container crashed earlier, or if Remove() was called twice by mistake, the directory might already be gone. Trying to delete something that doesn't exist causes an error.
The Solution: os.Stat checks if the folder exists. If it returns IsNotExist, we simply return nil (success). This makes the function idempotent (safe to call multiple times).

2. The "Zombie Process" Safety Net
The Issue: In Linux, you cannot delete a cgroup if it still has processes inside it. If your container process (/bin/sh) didn't exit cleanly (e.g., it became a "zombie" or got stuck), os.RemoveAll will fail with a "Device or resource busy" error.
The Solution:
    We read the cgroup.procs file, which lists the PIDs currently in that group.
    If the list is not empty, we print a Warning.
    Why not kill them here? In a simple runtime, we assume cmd.Wait() in container.go already handled the process exit. This check is primarily for debugging. If you see this warning, you know why cleanup failed.

3. The Recursive Delete
The Issue: A cgroup isn't just an empty folder. It contains many virtual files created by the kernel (like memory.max, cpu.stat, cgroup.procs).
    * os.Remove() only works on empty directories. It would fail immediately.
The Solution: os.RemoveAll() is recursive. It tells the kernel: "Delete this directory and everything inside it."
    * When the kernel sees this command, it destroys the cgroup hierarchy and frees the memory/CPU accounting structures associated with it.

The enableControllers Function

What it does: Intelligently updates the parent cgroup to allow specific resource controls.
How it works:
    It reads the existing cgroup.subtree_control file.
    It checks if +memory or +cpu is already there.
    It only writes the new ones.
Problem Solved: Idempotency & Race Conditions.
    * If you try to enable +memory when it's already enabled, some kernels throw an error.
    * If two containers start at the same time, they might both try to write to this file. Checking first reduces the chance of conflict.
CS Concept: State Management & Concurrency Safety:
Reading the current state before modifying it is a fundamental pattern in distributed systems and OS programming to prevent overwriting valid configurations.

    I also inspect cgroup.procs before deletion to detect zombie processes that might prevent cleanup,

1. "How does Linux isolate resources?"
Answer: "Linux uses two main kernel features: Namespaces for isolation (visibility) and Cgroups for limitation (accounting). My code implements Cgroups v2 by interacting with the sysfs virtual filesystem to set hard limits on Memory and CPU bandwidth."
2. "What is the 'Noisy Neighbor' problem and how did you solve it?"
Answer: "The Noisy Neighbor problem occurs when one tenant on a shared host consumes excessive resources, degrading performance for others. I solved this by setting cpu.max (CPU bandwidth throttling) and memory.max (hard memory caps) in the cgroup configuration."
3. "Why did you use SIGSTOP and SIGCONT in the container runtime?" (Connects to your container.go)
Answer: "To prevent a Race Condition. If I start the process and then apply cgroups, the process might execute instructions before the limits are applied. By sending SIGSTOP, I freeze the process, apply the cgroup limits safely, and then send SIGCONT to resume it. This ensures 100% of the process execution is governed by the limits."
4. "What is the difference between cgroups v1 and v2?"
Answer: "v1 had multiple hierarchies (one for memory, one for CPU), which was complex. v2 has a unified hierarchy. Also, v2 requires explicit delegation via cgroup.subtree_control, which is why I implemented the enableControllers function to write +memory and +cpu to the parent."
5. "What happens if a process exceeds memory.max?"
Answer: "The Linux Kernel's OOM (Out of Memory) Killer is triggered. Since I set memory.oom_group to 1, the kernel will kill all processes in that cgroup to protect the host system's stability."
